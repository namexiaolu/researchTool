# 个人知识库助手 — 分步实施指南

本指南把 `思路.md` 的三段方案拆成可执行的步骤，每一步说明**做什么、怎么做、为什么这样做**，并附上库的安装方式和作用介绍。

适用平台：Windows 11 + PowerShell（推荐 `pwsh` 7+）。Python 要求 3.10 及以上。

---

## 第 0 步：环境准备（一次性）

### 做什么
搭好 Python 虚拟环境，避免污染全局包。

### 怎么做
```powershell
cd D:\AI\mySkill
py -3.12 -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
```

如果激活脚本被策略拦截：
```powershell
Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned
```

### 为什么
- 虚拟环境把依赖锁在项目内，多项目不冲突，删项目即清理。
- 后续所有 `pip install`、`python xxx.py` 都在激活后的环境里跑。

## 第 1 步：安装核心库

按方案三段分别装。建议先装 RAG 段（最重，有 torch），再装 CLI 段。

### 1.1 调研段依赖（可选，先看说明再装）

```powershell
pip install httpx beautifulsoup4
```

- **httpx**：现代 HTTP 客户端，抓网页、调 Ollama 的 HTTP 接口都用它。支持同步/异步、超时可控，比 `requests` 更适合搭工具。
- **beautifulsoup4**：HTML 解析清洗库。抓回网页后用它剔除 `<script>`、`<nav>` 等噪音，只留正文文本。

> 调研段的"grok-search"和"K-Dense"按你现有的 MCP/插件方式调用，不需 pip 安装，本文不展开。

### 1.2 RAG 段依赖（关键）

```powershell
pip install langchain langchain-community chromadb sentence-transformers pypdf
```

逐个说明：

- **langchain**：RAG 编排框架本体，提供 Document 抽象、chain、LCEL 等核心原语。
- **langchain-community**：社区加载器/向量库适配集合，`PyPDFLoader`、`UnstructuredMarkdownLoader`、Chroma 适配器都在这里。
- **chromadb**：本地嵌入式向量数据库。数据落盘成 SQLite + parquet，零运维，个人规模够用。作用是存分块向量并做相似度检索。
- **sentence-transformers**：本地嵌入模型运行库。用它加载中文嵌入模型 `BAAI/bge-small-zh-v1.5`，把文本转向量。不依赖外部 API，隐私安全。
- **pypdf**：纯 Python 的 PDF 文本提取库，给 `PyPDFLoader` 当底层解析器。

> `sentence-transformers` 会自动带上 `torch`（深度学习框架，体积大，首次下载约 2GB）。属正常现象。

### 1.3 CLI 段依赖

```powershell
pip install typer rich
pip install langchain-ollama          # 接 Ollama（默认）
# pip install langchain-openai        # 想接云端模型时再加这行
```

- **typer**：基于类型提示的 CLI 框架。写一个带类型注解的函数就自动生成 `--help`、参数校验、子命令，比 argparse 省一半样板。
- **rich**：终端美化库，给 Typer 配套做彩色输出、表格、进度条。问答结果展示用得上。
- **langchain-ollama**：LangChain 对 Ollama 的官方封装，用类型化的 `OllamaLLM` 替代手写 httpx 调 `/api/chat`。

### 1.4 验证安装
```powershell
python -c "import langchain, chromadb, sentence_transformers, typer; print('OK')"
```
输出 `OK` 即装好。

## 第 2 步：安装并启动 Ollama（生成后端）

### 做什么
装 Ollama 并拉一个本地模型，作为问答的生成引擎。

### 怎么做
1. 到 https://ollama.com 下载 Windows 安装包，一路下一步（默认装到 `C:\Users\<你>\AppData\Local\Programs\Ollama`）。
2. 安装后系统托盘会出现 Ollama，后台已自动起 `http://localhost:11434`。
3. 拉模型：
```powershell
ollama pull qwen3:0.6b
```
拉完用 `ollama list` 确认。

### 为什么这样做
- **本地运行**：数据不出本机，适合个人隐私知识库；零 API 费用。
- **选 qwen3:0.6b**：600M 参数，普通笔记本 CPU 也能秒答；后续觉得能力不够再换 `qwen3:4b` 或 `llama3.2:3b`。
- **端口 11434**：Ollama 默认监听地址，后面 CLI 配置里要对齐。

## 第 3 步：建知识库目录

### 做什么
按 `思路.md` 的结构建目录并放料。

### 怎么做
```powershell
mkdir knowledge\sources, knowledge\papers, knowledge\web
```
然后把采集成果分别放进去：
- `knowledge\sources\links.md` —— 调研得到的链接 + 一句话简介（手写或脚本生成）。
- `knowledge\papers\*.pdf` —— 下载的期刊论文。
- `knowledge\web\*.md` —— 抓取清洗后的网页正文。

### 为什么这样做
- **分类落盘**：不同来源分开存，加载时按文件类型选 loader，元数据可追溯来源，问答时能告诉用户「这条答案来自哪篇论文」。
- **web 用 markdown 而非 html**：去掉网页骨架后只存正文，索引更干净，检索噪音小。

## 第 4 步：实现 RAG 索引（indexer/）

### 做什么
写 `indexer/` 三个文件：loaders、splitter、store，组合出"扫描→加载→分块→嵌入→入库"流程，并暴露 `build / update / list` 命令。

### 怎么做（骨架，按需实现）

**indexer/splitter.py**
```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

def get_splitter(chunk_size=512, overlap=64):
    # 中文友好的递归分块：优先按段落，再按句号、空格
    return RecursiveCharacterTextSplitter(
        chunk_size=chunk_size,
        chunk_overlap=overlap,
        separators=["\n\n", "\n", "。", "；", " ", ""],
    )
```

**indexer/loaders.py**
```python
from pathlib import Path
from langchain_community.document_loaders import PyPDFLoader, UnstructuredMarkdownLoader

def load(path: Path):
    ext = path.suffix.lower()
    if ext == ".pdf":
        return PyPDFLoader(str(path)).load()
    if ext in (".md", ".txt"):
        return UnstructuredMarkdownLoader(str(path)).load()
    return []
```

**indexer/store.py**
```python
from langchain_community.vectorstores import Chroma
from langchain_huggingface import HuggingFaceEmbeddings

VSTORE_DIR = "knowledge/vector_store"

def get_embeddings():
    return HuggingFaceEmbeddings(model_name="BAAI/bge-small-zh-v1.5")

def build(docs):
    vs = Chroma.from_documents(docs, get_embeddings(), persist_directory=VSTORE_DIR)
    return vs

def get_retriever(top_k=5):
    vs = Chroma(persist_directory=VSTORE_DIR, embedding_function=get_embeddings())
    return vs.as_retriever(search_kwargs={"k": top_k})
```
> `langchain_huggingface` 需 `pip install langchain-huggingface`。

### 为什么这样做
- **递归分块而非定长切割**：`RecursiveCharacterTextSplitter` 尽量在自然边界切，避免把一个完整论点拦腰截断，语义完整度更高。
- **嵌入用 bge-small-zh**：中文检索效果好，模型只有 ~100MB，首次下载快、推理省内存；不调外部 API。
- **Chrome 持久化**：建一次索引后续复用，无需每次问答重建，`persist_directory` 落盘保证重启不丢。
- **保留 metadata**：每个分块带 `source`，问答时可回溯出处，满足"能说出根据"的要求。

### 增量更新思路
记录每个文件的 hash 到 `knowledge/vector_store/.filehash.json`，`update` 时只对 hash 变化或新增的文件重新嵌入并 upsert，避免全量重建。

## 第 5 步：实现 CLI 问答（cli/）

### 做什么
用 Typer 写 `cli/main.py`，提供 `ask / query / chat / config / pull` 命令，组装"检索 + 生成"问答链。

### 怎么做（骨架）

**cli/main.py**
```python
import typer
from rich import print as rprint
from indexer.store import get_retriever
from langchain_ollama import OllamaLLM
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnablePassthrough

app = typer.Typer()

def build_chain(top_k=5):
    retriever = get_retriever(top_k)
    llm = OllamaLLM(model="qwen3:0.6b")
    prompt = ChatPromptTemplate.from_template(
        "你是知识库助手。基于下面上下文回答，不足就说不知道，不要编造。\n\n"
        "上下文：\n{context}\n\n问题：{question}\n\n回答："
    )
    chain = (
        {"context": retriever, "question": RunnablePassthrough()}
        | prompt
        | llm
    )
    return chain, retriever

@app.command()
def ask(question: str, verbose: bool = False):
    chain, retriever = build_chain()
    answer = chain.invoke(question)
    rprint(f"\n[bold cyan]回答[/]：\n{answer}\n")
    if verbose:
        docs = retriever.invoke(question)
        rprint("[dim]--- 参考来源 ---[/]")
        for i, d in enumerate(docs, 1):
            rprint(f"{i}. {d.metadata.get('source')}")

if __name__ == "__main__":
    app()
```
运行：
```powershell
python cli/main.py "RAG 的核心步骤是什么" --verbose
```

### 为什么这样做
- **Typer 而非 argparse**：函数签名即命令定义，`verbose: bool = False` 自动变 `--verbose` 开关，可读性和维护性都更好。
- **LCEL 管道组装链**：`retriever | prompt | llm` 这种直写式管道，比 `RetrievalQA.from_chain_type` 透明，出错好调，也方便后面插 MMR 重排。
- **rich 输出**：终端上彩色区分"回答/来源/错误"，长输出更易读，零额外成本。

### Ollama 连不上时
- 确认托盘有 Ollama，或手动 `ollama serve`。
- `ollama list` 看模型名是否与代码一致；不一致把 `OllamaLLM(model=...)` 改对，或 `ollama pull <名字>`。

## 第 6 步：贯通联调

### 做什么
跑通完整链路：放料 → 建索引 → 提问。

### 怎么做
```powershell
$ErrorActionPreference = 'Stop'
Set-Location -LiteralPath 'D:\AI\mySkill'

# 1. 放几个测试文件到 knowledge/papers 和 knowledge/web

```

### 验证清单
- [ ] 命令能跑，无 ImportError。
- [ ] 首次建索引 bge 模型下载成功（看进度条）。
- [ ] 提问后返回答案，且 `--verbose` 能列出来源文件名。
- [ ] 故意问知识库没有的内容，模型应回答"不知道"而非编造（靠 prompt 约束）。

- 先跑通最小闭环，再谈增量更新、MMR 重排、多轮 chat 等增强功能，避免一上来被细节卡住。

## 第 7 步：当前实现与维护说明

本节记录实际代码与前面方案骨架之间的差异，供后续开发维护使用。

### 7.1 实际命令入口

当前已经实现的入口只有：

- `python -m indexer.build_index`：扫描资料并写入 Chroma。
- `python -m cli.main "问题"`：检索并调用 Ollama 回答。当前只有一个 Typer 命令，运行时不写 `ask` 子命令名。

方案中的 `index build`、`index update`、`index list`、`query`、`chat` 和 `config` 尚未实现。

### 7.2 索引重建语义

`indexer/store.py` 当前直接调用 `Chroma.from_documents(..., persist_directory=...)`，不会先清空已有索引，也没有文件 hash 或增量更新逻辑。

因此维护资料后需要删除 `knowledge/vector_store/` 再全量构建。直接在旧索引上重复运行可能产生重复向量。后续应优先实现以下任一方案：

- 构建命令显式提供 `--rebuild`，先安全清理旧库。
- 为每个文件记录 hash，只更新新增或发生变化的文件。
- 为文本块生成稳定 ID，避免相同内容重复写入。

### 7.3 加载器依赖

当前 `.md` 和 `.txt` 都使用 `UnstructuredMarkdownLoader`，除 LangChain 外还依赖：

- `unstructured`
- `markdown`
- spaCy 的 `en_core_web_sm`

PDF 使用 `PyPDFLoader`，依赖 `pypdf`。如果只需要读取普通文本，后续可考虑复用 LangChain 的 `TextLoader`，减少 Markdown 解析器带来的额外依赖。

### 7.4 BGE 模型与离线运行

嵌入模型固定为 `BAAI/bge-small-zh-v1.5`，输出 512 维向量。模型下载完成后，建议开发和部署环境使用：

```powershell
$ErrorActionPreference = 'Stop'

$env:HF_HUB_OFFLINE = '1'
$env:TRANSFORMERS_OFFLINE = '1'
```

Windows 终端运行包含 `✓`、`✗` 等字符的脚本时，还应启用 UTF-8：

```powershell
$ErrorActionPreference = 'Stop'

$env:PYTHONUTF8 = '1'
[Console]::OutputEncoding = [System.Text.UTF8Encoding]::new()
```

### 7.5 构建验收

不能只看最后的“索引构建完成”。构建脚本会捕获单个文件的加载错误，然后继续处理其他文件。

验收时需要确认：

- 每个资料文件前都是 `✓`。
- 文本块数量大于 0。
- `chroma.sqlite3` 已生成。
- SQLite 中的向量数量符合分块数量。
- 使用 retriever 查询时可以返回正确来源。

### 7.6 后续迁移

当前使用的 `langchain_community.vectorstores.Chroma` 已被 LangChain 标记为弃用，但暂时不影响运行。后续维护时迁移到 `langchain-chroma`，并同步更新 `indexer/store.py` 的导入和初始化方式。

### 7.7 PowerShell 菜单脚本

项目根目录的 `启动项目.ps1` 是面向用户的统一入口，启动时自动激活 `.venv`，并提供调研、索引、报告和自由提问四个选项。

当前第一版的实现边界：

- 关键词调研复用 `.opencode/skills/myresearch/scripts/run_research.py` 生成调研计划，再由当前所选 AI 后端生成初步内容，不包含实时联网搜索。
- RAG 索引选项会校验目标路径，删除旧的 `knowledge/vector_store/` 后执行全量构建。
- 调研报告复用现有 RAG CLI，保存到 `knowledge/reports/`；需要重新构建索引后才会被后续检索使用。
- 自由提问调用 `python -m cli.main "问题" --verbose --provider <provider>`。
- 菜单 5 可以切换 `ollama`、`openai` 和 `ccswitch`，选择结果写入被 Git 忽略的 `.rag-settings.json`，其中不保存 API Key。
- 脚本支持 `pwsh -File .\启动项目.ps1 -SelfTest`，用于非交互检查环境和依赖状态。

### 7.8 OpenAI API 与 CCSwitch

`cli/llm_provider.py` 负责统一三种生成后端：

- `ollama`：使用 `langchain_ollama.OllamaLLM`。
- `openai`：使用 OpenAI Python SDK 的 Responses API，默认模型为 `gpt-5.6`。
- `ccswitch`：读取 CCSwitch 当前写入的 `~/.codex/config.toml` 和 `~/.codex/auth.json`，复用当前 Codex provider 的 `model`、`base_url`、`wire_api` 和 `OPENAI_API_KEY`。

CCSwitch 接入不读取或修改 `cc-switch.db`，也不会把密钥写入项目文件。当前第一版只支持 `wire_api = "responses"` 的 provider。

CLI 示例：

```powershell
$ErrorActionPreference = 'Stop'

python -m cli.main "问题" --provider ollama
python -m cli.main "问题" --provider openai
python -m cli.main "问题" --provider ccswitch
```

直接使用 OpenAI 后端时支持以下环境变量：

- `OPENAI_API_KEY`
- `OPENAI_MODEL`，默认 `gpt-5.6`
- `OPENAI_BASE_URL`，留空时使用 OpenAI 官方地址
- `OPENAI_REASONING_EFFORT`，默认 `low`

CCSwitch 路径可通过 `CCSWITCH_CODEX_CONFIG` 和 `CCSWITCH_CODEX_AUTH` 覆盖。模型、地址和 reasoning effort 也可分别通过 `CCSWITCH_MODEL`、`CCSWITCH_BASE_URL`、`CCSWITCH_REASONING_EFFORT` 覆盖。

OpenAI 官方参考：

- [Text generation / Responses API](https://developers.openai.com/api/docs/guides/text.md)
- [Latest model guidance](https://developers.openai.com/api/docs/guides/latest-model.md)
