# 个人知识库助手 方案

三段式流程：**调研采集 → RAG 建索 → CLI 问答**，面向个人持续沉淀知识。技术栈：LangChain（RAG）+ Typer（CLI）+ Ollama（本地生成）。与现有 `rag/rag.py` 无关，属全新项目。

---

## 一、调研（信息采集）

目标：多来源搜集资料，整理到统一知识库目录。

### 1.1 采集渠道

| 渠道 | 工具 | 产物 |
|------|------|------|
| LLM 多轮搜索 | grok-search（多轮迭代查询） | 链接 + 简介，写入 `sources/links.md` |
| 用mcp broser—relay调用浏览器插件多引擎搜索 | google / bing / baidu / 搜狗，各跑多轮去重 | 链接 + 简介，追加到 `sources/links.md` |
| 期刊 / 论文 | K-Dense（或类似学术检索） | 论文元数据 + PDF，下载到 `papers/` |

### 1.2 采集策略

- **多轮迭代**：首轮粗搜 → 提取关键词 → 二轮细化 → 交叉补充，避免单轮覆盖不全。
- **去重**：按 URL 归一化（去 query 参数、统一协议）后去重，相同主题合并。
- **分级标注**：每条记录附 `来源引擎`、`采集时间`、`相关度`，便于后续筛选。

### 1.3 落盘结构

```
knowledge/
├── sources/
│   └── links.md          # 链接 + 简介，Markdown 表格或列表
├── papers/                # 期刊论文 PDF 原件
│   └── *.pdf
└── web/                   # 已抓取正文（给 RAG 用）
    └── *.md
```

- **链接和简介**：统一写入 `sources/links.md`，含标题、URL、一句话摘要、来源、时间。
- **期刊论文**：下载 PDF 到 `papers/`，文件名用「作者-年份-短标题」。
- **网页正文**：抓取后清洗存为 `web/*.md`，剔除 script/nav 等噪音。

## 二、读取信息生成 RAG

目标：把采集到的 PDF / Markdown 索引进向量库，供问答检索。**框架选 LangChain**（简洁成熟，加载器/分块/检索组件齐全，PDF 解析开箱即用）。

### 2.1 技术选型

| 角色 | 选型 | 说明 |
|------|------|------|
| 文档加载 | `langchain-community.document_loaders` | `PyPDFLoader` 处理论文，`UnstructuredMarkdownLoader` 处理 web/、links.md |
| 文本分块 | `RecursiveCharacterTextSplitter` | 中文友好，按段落+句子递归切分，`chunk_size=512, overlap=64` |
| 嵌入模型 | `sentence-transformers` | 本地 `bge-small-zh-v1.5`（中文效果好、体积小），不依赖外部 API |
| 向量库 | `Chroma`（持久化） | 本地 SQLite 落盘，零运维，适合个人知识库规模 |
| 检索 | LangChain `Retriever` | 默认相似度 top_k=5，预留 MMR 重排接口 |

### 2.2 索引流程

1. 扫描 `knowledge/papers/`、`knowledge/web/`、`knowledge/sources/links.md`。
2. 按文件类型选择 loader → 解析为 Document（metadata 含 `source`、`file_type`）。
3. `RecursiveCharacterTextSplitter` 分块，保留原 source 元数据。
4. 嵌入 + 写入 Chroma 持久化目录 `knowledge/vector_store/`。
5. 增量更新：记录文件 hash，仅对新/改文件重新嵌入，避免全量重建。

### 2.3 命令

```
index build          # 全量构建/重建索引
index update         # 增量更新（对比 hash）
index list           # 列出已索引文档与块数
index add-web <url>  # 抓取网页清洗入库
```

### 2.4 依赖（精简）

```
langchain>=0.3
langchain-community>=0.3
chromadb>=0.5
sentence-transformers>=3.0
pypdf>=5.0
beautifulsoup4>=4.12
```

## 三、CLI 问答

目标：终端交互式问答，**检索知识库 + 本地 LLM 生成**，支持挂接 Ollama。框架选 **Typer**（类型提示自动生成 CLI，比 argparse 简洁，比 Click 更省样板）。

### 3.1 技术选型

| 角色 | 选型 |
|------|------|
| CLI 框架 | `typer`（+ `rich` 美化输出） |
| 生成后端 | Ollama（默认，本地隐私），可切 OpenAI 兼容 API |
| LangChain 接入 | `langchain_ollama.OllamaLLM` / `langchain_openai.ChatOpenAI` |
| 链组装 | `RetrievalQA.from_chain_type` 或 LCEL：retriever + prompt + llm |

### 3.2 命令

```
kb ask "问题"                # 基于知识库问答（检索+生成）
kb ask "问题" --verbose     # 同时打印参考来源与分数
kb query "问题"              # 仅检索，输出 JSON（调试用）
kb chat                      # 连续多轮对话（带历史）
kb config show               # 查看配置
kb config set top_k 8        # 修改配置
kb pull qwen3:0.6b           # 通过 Ollama 拉模型
```

### 3.3 问答链

1. 用户提问 → 嵌入 → Chroma 检索 top_k 个相关块。
2. 拼 prompt：系统提示（「基于上下文回答，不足就说不知道，不编造」）+ 检索上下文 + 问题。
3. 调 Ollama 本地模型生成回答，输出到终端。
4. `--verbose` 时附带来源文件名与相似度分数。

### 3.4 Ollama 接入要点

- 默认 `ollama_url=http://localhost:11434`，模型 `qwen3:0.6b`（轻量）；可选 `llama3.2:3b`、`qwen3:4b` 等。
- 启动前检测：连不上时提示「安装 Ollama / `ollama serve` / `kb pull <model>`」。
- 可切 OpenAI 兼容 API（base_url + key），方便接云端模型兜底。

### 3.5 依赖（精简）

```
typer>=0.12
rich>=13.0
langchain-ollama>=0.2         # 或 langchain-openai
httpx>=0.27
```
（嵌入与向量库依赖沿用第一节）

## 五、决策小结

- **RAG 框架**：LangChain —— 组件齐全、中文生态成熟，避免手搓 loader/splitter。
- **CLI**：Typer —— 类型提示驱动，命令即函数，比 argparse 省一半样板。
- **生成后端**：Ollama 默认本地，保护隐私、零 API 费用；保留 OpenAI 兼容接口兜底。
- **落地形态**：详细方案文档先行，后续按目录骨架逐步实现。
